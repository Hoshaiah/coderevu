## Context

This code lives in `auth/PasswordReset.php` and handles the one-time token verification step of a password-reset flow. Tokens are generated by hashing a random string with `md5()` and stored in the `password_resets` table. The endpoint is publicly accessible — no session is required to reach it.

During an internal red-team exercise, an engineer discovered they could complete the password reset for several accounts without knowing the correct token. The accounts affected all happened to have tokens that started with `0e` followed entirely by digits. The team initially suspected a database collation bug and spent two days checking the schema before turning back to the PHP code.

All database queries have been verified as correct: the stored hash is fetched accurately and matches what was emailed to the user. The problem is in how the comparison is performed after retrieval.

## Buggy code

```php
<?php
// auth/PasswordReset.php

require_once 'db.php';

$token    = $_GET['token'] ?? '';
$email    = $_GET['email'] ?? '';

$stmt = $pdo->prepare("SELECT token_hash FROM password_resets WHERE email = ? AND expires_at > NOW()");
$stmt->execute([$email]);
$row = $stmt->fetch(PDO::FETCH_ASSOC);

if (!$row) {
    http_response_code(400);
    exit('Invalid or expired reset link.');
}

$expected = $row['token_hash'];
$provided = md5($token);

if ($provided == $expected) {
    // Grant password reset
    $_SESSION['reset_email'] = $email;
    header('Location: /reset/new-password.php');
    exit;
}

http_response_code(403);
exit('Token mismatch.');
```
