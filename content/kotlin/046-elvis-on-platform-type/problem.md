## Context

`UserMapper.kt` converts a Java `UserRecord` (generated by a JOOQ schema) into a Kotlin domain model. The Java class has no nullability annotations. The team wrote defensive Kotlin code using the Elvis operator to handle missing display names.

In production, a `NullPointerException` is thrown inside `mapToUser` when a `UserRecord` row has a null `email` column. The stack trace points to the `User(...)` constructor call, not to any explicit dereference. The display-name mapping right above it is never reported as the problem.

The team confirmed that `getDisplayName()` itself works fine — the crash only happens on `email`. They assume the Elvis on `displayName` proved their null-handling pattern was correct and applied the same assumption to `email`.

## Buggy code

```kotlin
// UserRecord is a Java class with no nullability annotations:
// public class UserRecord {
//     public String getDisplayName() { ... }
//     public String getEmail() { ... }
// }

data class User(val displayName: String, val email: String)

fun mapToUser(record: UserRecord): User {
    val displayName = record.displayName ?: "Anonymous"
    val email = record.email  // platform type String! — no null check

    return User(
        displayName = displayName,
        email = email
    )
}
```
