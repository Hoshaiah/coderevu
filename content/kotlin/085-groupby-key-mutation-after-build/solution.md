## Reference solution

```kotlin
// ------------------------------------------------------------------------
// ANSWER — Grouped Map Keys Mutated After Build
// ------------------------------------------------------------------------

data class DimensionKey(val tags: List<String>)

class EventGrouper {

    private val groups = mutableMapOf<DimensionKey, MutableList<Event>>()

    fun add(event: Event, tags: MutableList<String>) {
        // CHANGE 1: Copy the list into an immutable snapshot so mutations by the caller after this call cannot change the key's hashCode or equals result.
        val key = DimensionKey(tags.toList())
        groups.getOrPut(key) { mutableListOf() }.add(event)
    }

    fun getGroup(tags: List<String>): List<Event> {
        // CHANGE 2: Defensively copy here too so a mutable list passed to getGroup cannot alias a stored key and cause instability during the lookup itself.
        return groups[DimensionKey(tags.toList())] ?: emptyList()
    }
}
```

## Explanation

### Issue 1: Mutable list stored by reference as map key

**Problem:** After `add()` returns, the caller mutates the `MutableList<String>` it passed in. Because `DimensionKey` stores that exact list reference, the `hashCode()` of the already-inserted key changes. HashMap buckets are chosen by hash, so the entry is now in the wrong bucket. Subsequent lookups — and even lookups by *other* keys — can silently return wrong results or find nothing.

**Fix:** At the `CHANGE 1` site, `tags.toList()` is called before constructing `DimensionKey`. This creates a new, independent `List<String>` snapshot whose contents cannot be changed by the caller.

**Explanation:** A `HashMap` (and Kotlin's `mutableMapOf`) computes `hashCode()` at insertion time to choose a bucket. If the key object's `hashCode()` later returns a different value — because its internal state changed — the entry sits in the wrong bucket forever. A `get()` with an equal key computes the new hash, looks in a different bucket, and finds nothing. `data class` generates `hashCode()` by delegating to its fields, and `List.hashCode()` depends on the list's current contents, so any mutation of the underlying list after insertion corrupts the key. Copying with `toList()` breaks the reference link, so the stored key's hash is stable for the lifetime of the map entry.

---

### Issue 2: Mutable list passed to getGroup can alias a stored key

**Problem:** If a caller passes the same `MutableList` instance to both `add()` and `getGroup()`, and that list is mutated between the two calls, the lookup key diverges from the stored key even though they started equal. The lookup then misses the group entirely.

**Fix:** At the `CHANGE 2` site, `tags.toList()` is called before constructing the lookup `DimensionKey`, so the lookup key is a stable snapshot regardless of what the caller does to the list it passed in.

**Explanation:** `DimensionKey.equals()` (generated by `data class`) compares the `tags` lists structurally at the moment `equals` is called. If the caller mutates `tags` between calling `add()` and `getGroup()`, the lookup key and the stored key have different contents, so `equals` returns `false` and the map returns `null`. This is a narrower window than Issue 1 — it only matters when the same list object is reused — but it is still a real failure path. Copying on the way into `getGroup()` ensures the lookup key reflects the list contents at call time, not whatever they become later.
