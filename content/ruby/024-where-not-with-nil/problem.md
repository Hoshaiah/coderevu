## Context

`app/models/user.rb` defines a scope used by the admin dashboard to list all users who are not internal staff. The scope has been in place since the app launched and the dashboard has always shown a user count that seemed reasonable, so it was never questioned.

A data audit triggered by a compliance request revealed that several hundred users with a `NULL` value in the `internal` column were not appearing in the dashboard at all. These were users created before the `internal` column was added and whose rows were never backfilled. The business logic requires that users with `NULL` should be treated the same as `false` — i.e. shown in the external-facing list.

The SQL generated by ActiveRecord is the culprit. The developer expected `where.not(internal: true)` to match both `internal = false` and `internal IS NULL`, but that is not how SQL NULL semantics work.

## Buggy code

```ruby
class User < ApplicationRecord
  # Intended: all users who are NOT internal staff.
  # This should include users where internal is false OR null.
  scope :external, -> { where.not(internal: true) }
end

# In the admin controller:
class Admin::UsersController < ApplicationController
  def index
    @users = User.external.order(:created_at)
    # Users with internal = NULL are silently excluded from results.
  end
end
```
